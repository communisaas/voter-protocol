## 0. Pre-Launch Assumption (EXPLICIT)

**Commons has no production users as of the v1 -> v2 cutover date.** Every
`districtCredentials` row in the Convex deployment is a beta-tester or
internal-test credential that can safely be rotated without user-facing
coordination.

This assumption MUST be verified by ops before running the cutover script
(`commons/scripts/cutover-v1-credentials.ts --execute`). If a production
user exists at cutover time, the operator follows the additional steps in
`commons/docs/runbooks/V2-CREDENTIAL-CUTOVER.md` §3 (beta-tester
notification window, re-verify throttle relaxation, rollback procedure,
monitoring setup).

### Pre-Cutover Checklist

- [ ] Convex query confirms active-user count matches beta-tester roster.
- [ ] 72-hour notification window honoured for any non-internal testers.
- [ ] `/api/internal/emit-revocation` HTTP endpoint live on the Commons
      deployment and authorized on `RevocationRegistry`.
- [ ] Backup of `districtCredentials` taken immediately before cutover
      (`convex export`). Enables the rollback procedure.
- [ ] Monitoring dashboards pre-wired for `RevocationEmitted`,
      `RevocationBlockedSubmission`, and Convex `revocationStatus` transitions.
- [ ] On-call operator available for the 7-day post-cutover window.

### In-Flight v1 Proof Handling at Cutover

A live browser tab with a cached v1 proof that submits after the v1 verifier
is decommissioned will receive a structural rejection from the v2 verifier
(public-input count mismatch). The Stage 2.5 UI catches this and surfaces
`CREDENTIAL_MIGRATION_REQUIRED` to the user. Users see:

> Your proof credential is being rotated. Please re-verify to continue.

The message directs them to the `IdentityRecoveryFlow` with a
"one-time security upgrade" label so they understand why the prompt
appeared. This explicit detection path is implemented in
`commons/src/lib/components/template/ProofGenerator.svelte` (coordinate
with the parallel Stage 2.5 wiring).

## 1. Context

Stage 2 of the re-grounding / address-change work plan introduces two cryptographic changes that, together, close F1 (stale-proof replay) and F2 (district-hopping amplification):

1. **`action_domain` v2**: the preimage now includes `district_commitment` (see `CRYPTOGRAPHY-SPEC.md` §6.4.1). `protocol_version` changes from `"commons.v1"` to `"commons.v2"`. **Shipped in Stage 2** at the TypeScript layer (`action-domain-builder.ts`).
2. **Revocation nullifier set**: a new public input (`revocation_nullifier`) at index 31 of the `three_tree_membership` circuit, checked against an on-chain `RevocationRegistry` (see `REVOCATION-NULLIFIER-SPEC.md`). **Design-only in Stage 2**; circuit + contract revision is a separate wave.

This migration plan covers the operational path from the current live state (v1 circuit + v1 action domain) to the future state (v2 circuit + v2 action domain + revocation enforcement). It addresses the core tension:

**Credentials issued under the v1 action-domain composition produce proofs whose `action_domain` public input was computed from the v1 preimage. v2 verifier contracts will not accept those proofs, because the `actionDomain` public input does not match any v2-whitelisted domain.**

Two strategies are considered. We recommend Option A.

---

## 2. Credential Compatibility — The Core Problem

A credential is *data*, not a proof. Credentials stored in `districtCredentials` (Convex) contain:

- `districtCommitment` (the sponge-24 output)
- `credentialHash` (Ed25519 signature over the attestation)
- 24-slot district IDs
- Validity metadata (`issuedAt`, `expiresAt`, `revokedAt`)

Credentials are **forward-compatible** with the v2 action-domain format: a pre-revision credential carries all the data a v2 prover needs to produce a valid v2 action domain (specifically, `districtCommitment`). The credential itself does not become stale on the v1 → v2 transition; only *proofs already generated* against a v1 action domain become inadmissible.

However, the revocation-nullifier public input *is* a circuit-level change. Proofs generated by the v1 circuit do not include a `revocation_nullifier` public input. A v2 verifier that expects 32 public inputs will reject v1 proofs structurally (different public-input count → verifier signature mismatch at the ACIR level).

So the two concerns are:

| Concern | Credential side | Proof side |
|---|---|---|
| v1 action domain | forward-compatible | proofs already generated are inadmissible under v2 verifier |
| Revocation nullifier | transparent to credential | v1 proofs physically cannot satisfy v2 public-input schema |

## 3. Option A — Revoke All Pre-Revision Credentials, Force Re-Verify (RECOMMENDED)

### 3.1 Summary

At cutover, all credentials issued before the v2 deployment are marked revoked. Users must re-verify their address to obtain a fresh credential. The on-chain revocation registry is pre-seeded with the revocation nullifiers for every pre-cutover credential. The v2 circuit and verifier are deployed atomically. v1 verifier contracts are decommissioned via the standard 7-day deactivation + 30–90 day grace period on root registries, which effectively retires v1 proofs from active routes.

### 3.2 Steps

1. **Pre-cutover: spec + circuit freeze.**
   - Complete the Stage 2 spec work (this wave, done).
   - Author the revised `three_tree_membership` Noir circuit per `REVOCATION-NULLIFIER-SPEC.md` §5.1.
   - Run the full CI: golden vectors, non-collision tests, Poseidon2 parity.

2. **Pre-cutover: contract preparation.**
   - Deploy `RevocationRegistry` (unused initially).
   - Deploy v2 HonkVerifier contracts for all four depth variants (18/20/22/24).
   - Deploy a revised `DistrictGate` (or a `DistrictGateV2` sibling) that wires in the revocation check and the 32-public-input verifier call.
   - Stage all deployments behind the 14-day verifier timelock; queue but do not execute.

3. **Cutover day — operator actions.**
   - **Step 3a** — Enqueue revocation nullifiers for every active pre-cutover credential:
     - Iterate `districtCredentials` where `revokedAt IS NULL AND expiresAt > now`.
     - For each: compute `revocation_nullifier = H2(districtCommitment, REVOCATION_DOMAIN)`.
     - Batch-submit to `RevocationRegistry.recordRevocation(…)`. Gas: cheap per-write on Scroll (~$0.00015); total cost for 100K credentials ~$15.
   - **Step 3b** — Patch Convex: set `revokedAt = now` on every previously-active credential. Single `patch` in an admin mutation.
   - **Step 3c** — Execute the queued verifier upgrades; `DistrictGateV2` becomes the active route.
   - **Step 3d** — Deactivate the old v1 `DistrictGate` (requires the standard deactivation timelock but is initiated immediately).

4. **Post-cutover: user re-verification.**
   - When users land at the submission flow, the Stage-3 UI (parallel track) detects `revokedAt` and pivots to the `IdentityRecoveryFlow` already wired into `VerificationGate`. Users re-verify; new credential carries a fresh `districtCommitment`; new action-domain-v2 proofs verify.
   - Re-verification throttle (24h + 6/180-day sybil cap — Stage 1) continues to apply, preventing runaway re-verification churn.

5. **Post-cutover: cleanup.**
   - After the 30–90 day root-deactivation grace period, the v1 `DistrictGate` is fully retired.
   - Sunset the v1 HonkVerifier contracts (free storage on Scroll indexers; contract bytecode is immutable).

### 3.3 User Impact

Every active user must re-verify once. Stage 1's rate-limited `verifyAddress` supports this: the 24h cooldown is per-user, not per-epoch, so a concurrent global re-verification push does not hit the throttle. The email-sybil cap (6 userIds per email hash per 180 days) is the only binding constraint and is sized above expected churn.

For users: one address-entry flow, completed in <60 seconds. For the operator: comms that the re-verification is a security upgrade, not a data request.

### 3.4 Pros

- **Simple invariant**: on cutover day, every pre-v2 proof is dead. No mixed-mode verifier logic. No code path has to reason about "is this a v1 or v2 proof."
- **Security-maximizing**: the window during which any stale proof remains admissible is zero.
- **Cheap**: ~$15 in Scroll L2 gas. No new governance primitives beyond what `RevocationRegistry` already needs.
- **Aligns with existing architecture**: root-deactivation grace periods exist exactly for this kind of clean break.

### 3.5 Cons

- **UX cost**: all active users must re-verify at once. If the user base is large (post-launch scale), this is a comms burden.
- **Operator attention required during cutover window**: re-verification volume spikes for ~1 week.

## 4. Option B — Legacy-Proof Acceptance Window

### 4.1 Summary

Deploy both v1 and v2 verifiers simultaneously behind a "mode" flag in `DistrictGate`. Accept v1 proofs against v1-whitelisted action domains (which do not bind district_commitment) for a bounded window (e.g., 90 days). During the window, new credentials are issued with both v1 and v2 action-domain acceptances; new proofs can be either. After the window, v1 verification is disabled via governance timelock.

### 4.2 Steps

1. Pre-cutover: same as Option A steps 1–2, plus:
   - Modify `DistrictGate.verifyThreeTreeProof` to accept both 31-input and 32-input public-input arrays; branch to the appropriate verifier.
   - Modify `action-domain-builder.ts` to return both v1 and v2 forms so the backend can whitelist both.
2. Cutover: enable dual-mode.
3. Over 90 days: naturally retire v1 as users re-verify for other reasons (address changes, expiring credentials). Emit deprecation warnings on v1 proof submissions.
4. End of window: revoke v1 `action_domain` whitelist entries via the standard 7-day revocation timelock; disable v1 verifier routing.
5. Only now: enqueue revocation nullifiers for any credential that *still* has not been re-verified. Continue as Option A step 4.

### 4.3 Pros

- **No forced user migration** during the window; natural re-verification does most of the work.
- **Lower cutover-day risk**: the v1 path remains live as a fallback if the v2 path has undiscovered issues.

### 4.4 Cons

- **F2 structural closure is not universal during the 90-day window.** v1-mode proofs still bind `action_domain` to the client-chosen `recipientSubdivision` only, so district-hopping amplification remains admissible for 90 days. This is a core security regression against the Stage 2 thesis.
- **F1 structural closure is also delayed.** Revocation nullifier enforcement only applies to v2 proofs during the window; v1 proofs generated against revoked credentials remain admissible to the v1 path.
- **Implementation complexity balloons**: dual verifiers, dual public-input schemas, dual action-domain whitelist entries, dual event schemas, dual test matrices. Every downstream consumer (scorecard, engagement tree update, accountability receipt) must handle both event shapes. Estimated 2–3 weeks of additional engineering over Option A.
- **Audit surface increases**: a mixed-mode gate is a classic source of spec-gap bugs. Every analysis has to say "and in mixed mode…".
- **Sunsetting risk**: "90 days" tends to slip. The security benefit of Stage 2 is only fully realized on the day the v1 path is off. Option B creates a temptation to keep the fallback live.

### 4.5 Why Not Recommended

The Stage 2 premise is that F1 and F2 are structural cryptographic gaps. A 90-day window where F1 and F2 remain *partially* open, in exchange for softer user UX, inverts the security posture. The win is marginal (users were going to re-verify anyway for natural reasons) and the cost is a non-trivial engineering build-out plus real residual attack surface.

Option B might be defensible if:

- The user base were vast (tens of millions), making a forced re-verify operationally impractical.
- There were mission-critical pending proofs that cannot afford a cutover blackout.

Neither applies pre-launch (Commons is pre-launch per the memory; first orgs have not onboarded). A forced-migration cutover now is cheap; later it will be expensive.

---

## 5. Recommendation: Option A

**Revoke all pre-revision credentials at cutover. Force re-verification.**

Reasoning:
- Pre-launch phase: forced re-verify has near-zero user-facing cost.
- Stage 2's stated goal is *structural* closure of F1/F2. Option B's 90-day window compromises that goal.
- Cutover engineering is ~1 week: a `RevocationRegistry` batch-write script, a Convex admin mutation, the verifier-registry upgrade. Option B is ~3 weeks of dual-path code paths.
- The clean-break cutover simplifies downstream analysis: one mental model for submissions, one event schema, one audit trail.

## 7. Risk Register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Cutover-day bug in the verifier upgrade | Low | High | Full Foundry test pass against testnet before executing timelock; queue a cancel tx alongside. |
| Revocation batch write partially fails | Medium | Medium | Script is idempotent (AlreadyRevoked reverts are treated as success). Re-run safely. |
| Users re-verify before v2 rollout is complete | Low | Medium | Stage 1 throttle enforces 24h cooldown; post-v2 re-verification uses the new credential automatically. |
| Stage 3 UI not ready for re-verify pivot | Medium | Medium | Cutover is coupled to Stage 3 readiness. Do not ship v2 until UI can express "one-time security upgrade." |
| Residual browser-cached v1 proofs attempt submission post-cutover | High | Low | Rejected cleanly by v2 verifier (public-input count mismatch). User sees error; prompts re-proof generation. |
| Unknown-unknown in v2 circuit | Medium | High | 14-day verifier timelock allows independent reproduction + bytecode comparison before activation. |

## 8. Rollback Plan

If a critical defect is discovered in the v2 circuit or contracts after cutover:

1. **Within the 7-day grace period on root deactivation**: re-activate the v1 `DistrictGate` via governance. Revert the `DistrictGateV2` activation. Note: users who re-verified during cutover now hold fresh credentials; they are fine either way because credentials are forward-compatible (see §2).
2. **After the grace period**: v1 `DistrictGate` is deactivated. Re-activation requires re-registering the v1 root and accepting the associated timelock windows. Slow but possible.
3. **Worst case — v2 circuit is unsound**: emergency pause `DistrictGateV2`. Users cannot submit proofs until the defect is patched and a new v2' circuit + verifier is deployed.

No rollback path re-admits stale v1 proofs after cutover; this is intentional. The security-regression cost is deemed greater than any UX cost of a pause.

---

## 9. Non-Goals

- **Not a credential format change.** `districtCredentials` schema is unchanged.
- **Not a key rotation.** Ed25519 signing key used for VC issuance is unchanged.
- **Not a user-secret rotation.** User secrets (Tree 1 leaf preimages) are unchanged; Tree 1 roots do not need to be rebuilt.
- **Not a Tree 2 / Tree 3 rebuild.** Cell-map and engagement trees are unchanged.

The only things that change: (a) the v2 `action_domain` composition, (b) the circuit's public-input layout, (c) the `DistrictGate` verification path, (d) one batch of on-chain revocation writes at cutover.
